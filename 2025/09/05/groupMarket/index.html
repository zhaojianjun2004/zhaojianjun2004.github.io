拼团系统是如何防止超卖的？揭秘“锁单”背后的核心设计(一)

发表于2025-09-05|更新于2025-09-05|架构设计电商系统

一、你以为的拼团，其实早就“满了”

你有没有遇到过这样的情况？

晚上8点，某平台爆款水果拼团开抢。你准时点开链接，手速飞快地点击“立即参团”，页面跳转到支付页，正准备扫码付款——

突然弹出一行字：

“抱歉，该团已满，请加入其他团或发起新团。”

你愣了一下：刚才不是还能点进去吗？怎么一转眼就满了？

更奇怪的是，你朋友却告诉你：“我刚拼上了啊，一切顺利。”
你们明明是同一时间操作的，为什么结果完全不同？

拼团，不是“谁先点谁赢”

很多人以为，拼团就像抢票，谁手速快，谁就能抢到。
但现实是：在你点击“参团”的那一刻，胜负早已决定。

真正决定你能不能参与的，不是你的手指有多快，而是系统是否还允许“新增一个参团名额”。

换句话说：

即使页面没灰显、按钮可点击，那个团，可能早就满了。

那个“看不见的门槛”：锁单机制

为什么会出现这种“能进不能成”的情况？

因为现代拼团系统里，有一个你看不见但至关重要的环节——

锁单（Lock Order）

它发生在你点击“参团”之后、支付之前，是系统为你临时锁定拼团资格和优惠价格的过程。

你可以把它理解成：

“先占坑，再付款”
坑占住了，你才有资格支付；
坑没了，哪怕你已经到了支付页，也会被拦下来。

举个生活化的例子

想象一下电影院抢《哪吒2》首映票：

  • 影院总共100个座位
  • 当98人已经下单未支付时，系统会把这98个座位“暂时锁住”
  • 第99个人来买票：还能买，系统说“请15分钟内付款”
  • 第100个人来买票：也可以，刚好凑满
  • 第101个人来买票：不行了，哪怕前面有人还没付钱，系统也说“售罄”

这就是“锁而不付，仍占资源”的典型场景。

拼团也是一样：

只要锁单人数 + 已支付人数 = 目标人数，新用户就再也无法加入。

所以，你看到的“还能参团”，可能只是假象

前端页面不会实时刷新每个团的精确状态（否则服务器早崩了），所以它往往只做“粗略判断”。

你点进去的时候，系统可能刚查到“还有名额”，页面就放行了。

可就在你浏览商品的几秒钟里，别人已经完成了锁单。

等你真正提交时，系统再查一次：“不好意思，现在满了。”

这不是系统有问题，恰恰说明它运行得太严谨了。

接下来，我们就来揭开这个“锁单”机制的神秘面纱——它到底锁了什么？又是怎么保证公平的？

二、“锁单”才是拼团真正的入场券

当你点击“立即参团”的那一刻，你以为自己已经成功加入了拼团。
但其实，系统才刚刚开始为你“申请资格”。

你还没有拼上，只是拿到了一张“准考证”——这张证，就是“锁单”。

什么是锁单？

简单来说，锁单 = 锁定拼团资格 + 锁定优惠价格 + 占用一个参团名额。

它不是支付，也不是成团，而是一个前置的、临时的、可撤销的承诺机制。

你可以把它理解为：

“系统答应你：只要你在规定时间内付款，就能享受拼团价，且这个位置不会给别人。”

但如果超时未付，系统就会收回承诺，把这个位置让给其他人。

为什么必须有锁单？直接支付不行吗？

听起来多此一举？我们来看几个现实问题：

问题 如果没用锁单 有了锁单
用户A点击参团，系统计算拼团价是9.9元，但支付时变成19.9元 ❌ 价格变动，用户体验差 ✅ 价格提前锁定
5人成团，已有5人锁单未付，第6人还能参团吗？ ❌ 可能导致6人“参团”，超卖 ✅ 检查锁单数，拒绝加入
网络卡顿，用户连点两次“参团” ❌ 生成两个订单 ✅ 幂等控制，只允许一个

👉 所以，锁单是拼团系统稳定运行的“安全阀”。
它把“资格确认”和“资金支付”拆成两个阶段，既保障用户体验，又防止资源错配。

锁单到底锁了哪些东西？

我们可以把一次锁单拆解为三个核心动作：

  1. 锁定优惠价格
    调用营销系统试算：原价59.9 → 拼团价9.9，这个价格在锁单时就定下来，后续不会变。
  2. 创建待支付订单（流水单）
    记录用户、商品、价格、拼团ID等信息，状态为“待支付”。
  3. 更新拼团人数（lock_count +1）
    在拼团主表中，将“已锁单人数”加1，表示有一个新用户占了坑。

这三件事必须在一个事务里完成：要么全部成功，要么全部回滚，否则就会出现“人锁了但没订单”或“订单有了但没名额”的脏数据。

📌 这条SQL的精妙之处：

  • 原子性：判断和更新一步完成
  • 并发安全：多个请求同时执行，只有一个能成功
  • 防超卖：基于“总占用人数”判断，不是只看 lock_count

如果更新影响行数为0，说明条件不满足 → 拼团已满 → 抛异常。

4. 举个例子：两人同时参团，谁赢？

假设一个5人拼团，当前状态：

  • complete_count = 3
  • lock_count = 1
  • 总占用 = 4 < 5 → 还剩1个名额

用户A和用户B同时发起参团请求：

  1. 数据库为 team_id 加行锁
  2. 第一个到达的请求（比如A）执行SQL：
    • (1 + 3) < 5 → 成立 → lock_count 变成2 → 成功
  3. 第二个请求（B）再执行：
    • (2 + 3) = 5 → 不小于 → 条件失败 → 更新0行 → 失败

结果：A成功锁单，B被拒绝。

✅ 公平、准确、无超卖。
